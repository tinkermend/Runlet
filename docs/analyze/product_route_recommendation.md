# Runlet 面向企业级自动化仿真巡检与测试平台的产品路线建议

## 1. 文档目的

本文基于当前 `Runlet` 项目的架构约束、已实现主链以及对 `Midscene`、`Playwright CLI`、`bb-browser`、`browser-use`、`Cypress`、`agent-browser` 的对比结果，回答三个核心问题：

1. 哪些能力应该自己做。
2. 哪些能力应该集成现成开源项目。
3. 哪些能力当前不该做。

本文视角明确限定为：

**目标产品是企业级自动化仿真巡检与测试平台，而不是通用浏览器 Agent。**

## 2. 当前目标再定义

如果把目标收敛成一句话，`Runlet` 当前最合理的产品定位是：

**面向企业后台系统的资产化、可调度、可审计、服务端认证注入的自动化仿真巡检与测试平台。**

这一定义天然包含以下关键词：

- 企业后台
- 只读检查优先
- 资产优先
- control plane 统一编排
- 服务端统一认证
- 执行可复现
- 结果可审计
- 能从单次执行晋升为调度对象

这也意味着它不应该优先追求：

- 通用网页自由探索
- 纯自然语言驱动的一切任务
- 写操作主链
- 临场生成脚本作为唯一真相

## 3. 总体判断

### 3.1 应该自己做的

必须自己做的是：

- 平台真相层
- 业务治理层
- 资产模型层
- 认证治理层
- 调度与审计层

### 3.2 应该集成开源的

适合集成的是：

- 浏览器执行底座
- 调试与取证能力
- 辅助报告与可视化能力
- 前端自身测试框架

### 3.3 当前不该做的

当前不该做的是：

- 通用浏览器 Agent 化
- 写操作主链
- 本地登录态正式复用
- 无边界 DSL / 自由脚本平台

## 4. 哪些能力应该自己做

这一类能力直接决定 `Runlet` 是否还是 `Runlet`。如果把这些交给外部项目，产品定位会被冲掉。

### 4.1 Control Plane 与正式执行编排

必须自己做：

- `check_request` 受理
- 轨道选择（`precompiled` / `realtime_probe`）
- 正式执行授权判断
- 任务投递与执行状态流转
- 从执行结果晋升到调度对象

原因：

- 这是平台的唯一正式入口。
- 这是你系统里“谁能执行、执行什么、为什么能执行”的治理中心。
- 这也是企业产品与通用浏览器工具之间最本质的差异。

### 4.2 资产模型与命中链路

必须自己做：

- `intent_aliases`
- `page_assets`
- `page_checks`
- `module_plans`
- `asset_version`
- 漂移与退役治理

原因：

- 资产模型是平台壁垒，不是执行库壁垒。
- 只有你最理解什么叫“企业后台页面检查资产”。
- 一旦这层外包给通用 Agent 或自由脚本系统，后续的可复用、可回归、可调度都会失真。

### 4.3 服务端认证治理

必须自己做：

- 认证策略定义
- 认证刷新
- 服务端 storage state 管理
- session / PAT / channel-action-system 判权
- 正式执行链中的认证注入

原因：

- 企业执行平台最敏感的就是认证边界。
- 当前仓库已经明确禁止把本地浏览器登录态复用为正式执行主链。
- 这部分不能依赖浏览器 Agent 框架“顺手处理”。

### 4.4 结果模型与审计模型

必须自己做：

- `execution_request`
- `execution_plan`
- `execution_run`
- `execution_artifact`
- `published_job`
- `job_run`

原因：

- 企业要看的不是“脚本跑没跑”，而是“请求、计划、执行、证据、调度”是否完整闭环。
- 谁触发、跑了什么、失败在哪、截图在哪、是否能发布，这些都是产品能力，不是底层执行能力。

### 4.5 巡检模板与业务断言模板

必须自己做：

- `has_data`
- `no_data`
- `field_equals_exists`
- `status_exists`
- `count_gte`
- 后续企业后台常见只读模板族

原因：

- 这是贴近业务价值的一层。
- Playwright、browser-use、Midscene 都不能替你定义“企业业务里哪些断言最值钱”。
- 这是未来形成平台标准能力包和行业模板库的核心。

### 4.6 调度治理与运行策略

必须自己做：

- `published_job`
- `runtime_policy`
- `auth_policy`
- `crawl_policy`
- 调度启停、回流、暂停、退役联动

原因：

- 巡检平台的复利不在“跑一次”，而在“长期稳定运行”。
- 调度对象必须绑定资产版本、认证策略和运行策略，不能退化成一段孤立脚本文本。

### 4.7 Drift / Compile / Reconciliation 主链

必须自己做：

- crawl 事实采集
- asset compile
- reconciliation
- retired / disabled / paused cascades

原因：

- 企业后台会变化，平台必须知道“什么时候旧资产已经不可信”。
- 这套机制决定正式执行是否仍然安全、是否仍然可复用。

## 5. 哪些能力应该集成现成开源项目

这一类能力更像“底座”或“辅助层”，重复造轮子收益不高。

### 5.1 浏览器执行底座

建议集成或持续吸收：

- 官方 `Playwright`
- `Playwright CLI`
- 局部能力上参考 `agent-browser`

适合集成的能力：

- trace / video / screenshot / PDF
- storage state / cookies / localStorage / sessionStorage 管理
- network / HAR / request interception
- page snapshot / console / DOM diagnostics
- 更丰富的命令式浏览器操作能力

建议定位：

- 作为 `runner_service` 的执行底座增强
- 作为研发调试与运维诊断能力

不建议定位：

- 不要替代 `control_plane`
- 不要替代资产模型

### 5.2 调试与取证增强能力

建议优先吸收：

- `Playwright CLI`
- `agent-browser`

适合集成的能力：

- step 级 trace
- HAR 导出
- network diff
- console / page errors 聚合
- 更好的浏览器会话观察能力

原因：

- 当前 `Runlet` 已有截图与结构化步骤结果，但 step 级取证还不够强。
- 这部分做强后，能显著提高排障效率和结果可信度。

### 5.3 可视化报告与视觉辅助能力

建议选择性吸收：

- `Midscene`

适合集成的能力：

- 更好的执行报告呈现
- 页面视觉对照
- 某些 DOM 不稳定页面的受控视觉辅助定位

建议定位：

- 报告层
- 诊断层
- 受控 fallback

不建议定位：

- 不要让视觉模型成为正式执行的系统真相

### 5.4 前端自身测试框架

建议直接使用现成方案：

- `Cypress`
- 或官方 `Playwright Test`

适用范围：

- `Runlet` 自己的管理平台前端
- 控制台 E2E
- 组件测试
- 前后端联调回归

原因：

- 这是成熟问题，不值得自己造轮子。
- 但这套测试框架属于 `Runlet` 自己的研发测试工具链，不属于平台对外正式执行主链。

### 5.5 特殊站点用户态接入思路

建议参考但谨慎使用：

- `bb-browser`

适合借鉴：

- 复杂登录站点接入验证
- 用户态网站转 API/CLI 的思路
- 调试链路和研究工具

不建议纳入：

- 企业正式执行主链

原因：

- `bb-browser` 的优势建立在“复用真实用户浏览器登录态”上。
- 这和当前项目“服务端统一认证注入”的正式边界天然冲突。

### 5.6 通用 Agent 辅助能力

建议有限参考：

- `browser-use`

适合借鉴：

- human-in-the-loop
- cloud browser / profile 管理
- 多工具编排体验

建议定位：

- 研发探索工具
- 辅助分析链路
- 非正式执行链路

## 6. 哪些能力当前不该做

这一部分最重要，因为很多方向“看起来先进”，但会直接破坏产品定位。

### 6.1 不要做成通用浏览器 Agent

当前不该做：

- 任意网站自由探索
- 临场动态规划复杂流程
- 一句话驱动跨站操作
- 无限制网页任务代理

原因：

- 这会把产品从“企业治理平台”拉向“通用 Agent”。
- 方向一旦偏掉，资产模型、控制面、认证治理都会被边缘化。

### 6.2 不要把自由脚本或自由 DSL 作为系统真相

当前不该做：

- 让自由生成的 Playwright 脚本成为正式执行主链
- 过早引入无限制 DSL
- 允许 `runner_service` 绕过资产层直接解释复杂自然语言

原因：

- 当前的长期优势来自 deterministic path。
- 自由脚本更适合作为派生产物、调试产物、研发工具，而不是系统真相。

### 6.3 不要把写操作作为主线

当前不该做：

- 创建
- 删除
- 审批
- 支付
- 提交生产数据变更
- 跨系统事务式业务流

原因：

- 写操作会显著提高风控、权限、回滚和审计复杂度。
- 当前最值得做深的仍然是只读巡检、结构化断言、页面级仿真验证。

### 6.4 不要让本地浏览器态进入正式执行主链

当前不该做：

- 直接复用个人 Chrome 登录态作为正式执行主链
- 把浏览器插件或本地 profile 作为正式认证真相

原因：

- 与当前安全边界冲突。
- 难以统一治理、轮转、审计和服务端托管。

### 6.5 不要过早做跨页面复杂业务编排引擎

当前不该做：

- 类 BPM 的复杂流程执行引擎
- 大规模多页面状态机编排
- 自由组合跨页面业务动作

原因：

- 当前资产模型、模板模型、调度模型更适合先把单页只读检查做到高覆盖。
- 过早做复杂流程会让问题空间急剧膨胀。

### 6.6 不要为了“更智能”引入过多黑盒 AI 决策

当前不该做：

- 用大模型替代资产命中主链
- 用黑盒视觉模型替代核心断言
- 用 LLM 直接决定正式执行步骤而缺乏确定性约束

原因：

- 在企业巡检产品里，可解释、可审计、可复现通常比“更像 Agent”更重要。

## 7. 推荐的近期路线

## 7.1 第一优先级：把平台主链做深

建议继续加强：

- 资产命中率
- 模板断言覆盖率
- 执行证据模型
- 调度治理
- 退役/漂移治理
- 认证治理

目标：

- 把“企业后台常见只读巡检”做到高覆盖、高稳定、高可解释。

## 7.2 第二优先级：增强执行底座与诊断体验

建议投入：

- Playwright trace / HAR / video
- 更强的 network / console / error diagnostics
- 更好的运行报告与证据聚合
- 研发侧调试工具链

目标：

- 让平台不仅能跑，还要更容易定位“为什么失败”。

## 7.3 第三优先级：有限引入 AI 辅助

建议探索：

- Chat 入口的模板推荐更智能
- 候选排序更智能
- 视觉辅助报告
- 研发态 AI 排障辅助

目标：

- 让 AI 优先增强“入口与辅助层”，而不是替代“正式执行真相层”。

## 8. 最终建议

可以把 `Runlet` 的能力边界概括成一句话：

### 8.1 自己做的

自己做：

- 控制面
- 资产模型
- 认证治理
- 调度治理
- 审计模型
- 模板断言

### 8.2 集成的

集成：

- 浏览器执行底座
- 调试与取证能力
- 可视化报告能力
- 前端自身测试框架

### 8.3 不做的

不做：

- 通用浏览器 Agent
- 写操作主链
- 本地登录态正式复用
- 自由脚本/DSL 真相化

## 9. 结论

`Runlet` 最有价值的路线，不是“我也能让 AI 去点网页”，而是：

**我能把企业后台的巡检与测试能力沉淀为资产，统一受理、统一认证、统一执行、统一审计，并长期调度运行。**

这条路线下：

- `Playwright CLI / agent-browser` 更像你应该吸收的执行底座。
- `Midscene` 更像你可以借鉴的视觉辅助与报告能力。
- `Cypress` 更适合你自己的前端测试工具链。
- `bb-browser / browser-use` 更适合作为研究参考或辅助链路，而不是正式主链。
