# 面向客户的 Web 前端管理平台一期设计

**日期：** 2026-04-04  
**作者：** Codex  
**状态：** Draft

---

## 1. 文档定位

本文档定义当前项目基于既有后端能力建设 Web 前端管理平台第一阶段的设计方案，目标是为客户侧普通使用者提供一个可登录、可配置检查任务、可查看采集资产与运行结果的管理后台。

本设计明确建立在当前后端正式链路之上：

- 检查资产是主模型
- Playwright 脚本是派生产物
- 正式执行统一走 `control_plane`
- 认证注入必须由服务端统一处理

因此，本次前端设计不是新增一套独立执行逻辑，而是把现有 `auth / crawl / asset compile / page_check / published_job / runtime policy` 能力产品化、客户化。

---

## 2. 背景与范围判断

### 2.1 已确认的产品前提

本轮讨论已确认以下约束：

1. 前端第一阶段的主要目标用户是**客户侧普通使用者**
2. 用户进入系统后的第一核心任务是：**配置检查任务并定时运行**
3. 第一版不做复杂 RBAC，只提供一个简单 Web 登录
4. 前端需要支持基础“系统接入”表单，但接入仍由后端受控执行
5. 采集数据需要同时支持：
   - 默认业务友好视图
   - 详情页查看原始事实
6. 技术选型采用：
   - `React + Vite`
   - 不使用重型后台脚手架
   - 采用轻量自建管理台

### 2.2 为什么这不是普通 Admin 后台

本项目的一期前端虽然是管理后台形态，但核心用户目标并不是表格 CRUD，也不是权限配置，而是：

- 接入自己的业务系统
- 基于已采集资产配置检查任务
- 将检查任务发布为定时运行
- 查看执行结果并定位问题

这决定了一期前端必须围绕“任务向导 + 任务运营 + 结果查看”展开，而不是照搬传统 `菜单管理/角色管理/字典管理` 模板。

### 2.3 一期范围判断

一期前端只解决“让客户能够用起来”的闭环，不追求一次性覆盖全部平台治理能力。

---

## 3. 设计目标与非目标

### 3.1 设计目标

本阶段前端平台需要实现以下目标：

1. 为客户提供简单登录后的统一 Web 管理入口
2. 让客户可以通过**任务向导**创建检查任务，而不是直接理解底层 `page_check / published_job`
3. 让客户可以查看自己已接入的系统、可用页面资产、可配置检查项
4. 让客户可以配置并管理检查任务的调度运行
5. 让客户可以查看最近运行结果与失败摘要
6. 为未来多团队/多客户扩展预留信息架构边界，但一期不引入复杂租户与权限体系

### 3.2 非目标

本次设计明确不做以下内容：

- 不做复杂 RBAC 或细粒度角色权限管理
- 不做多租户强隔离方案落地
- 不做自由 Playwright 脚本编辑器
- 不把 `page_check / published_job / script_render` 直接作为面向客户的主心智
- 不把认证态、`storage_state`、底层浏览器执行细节暴露给客户
- 不在前端绕过后端 `control_plane` 直接拼接正式执行逻辑
- 不把系统治理、策略治理、内部运维能力一次性全部搬到客户端主导航

## 4. 方案比较与推荐

### 4.1 方案 A：任务中心型前端

做法：

- 把“检查任务”作为前端第一主对象
- 首页、导航和主流程都围绕“创建任务、管理任务、查看结果”展开
- 采集资产和系统接入作为支撑模块存在

优点：

- 最符合客户侧普通用户的目标
- 学习成本最低
- 能把底层平台复杂度收敛在向导与详情页中

缺点：

- 底层资产模型和调度能力不会完整暴露在主导航中
- 高级平台治理能力需要放到二级页面或后续版本

### 4.2 方案 B：调度运营型前端

做法：

- 把调度计划、运行窗口、执行记录和策略状态放在主导航前排
- 用户主要从调度视角理解平台

优点：

- 便于平台运营与内部运维使用
- 任务执行与异常处理能力较突出

缺点：

- 对客户侧普通使用者不够直观
- 容易让产品心智退化为“平台运维台”

### 4.3 方案 C：资产浏览型前端

做法：

- 把系统、页面资产、检查项和原始事实作为前端第一主对象
- 用户先理解采集成果，再从资产反向创建任务

优点：

- 资产结构最清楚
- 适合资产治理型用户

缺点：

- 过早暴露平台底层模型
- 对普通客户的上手门槛偏高

### 4.4 推荐结论

采用 **方案 A：任务中心型前端**，并吸收少量方案 B 的调度运营能力：

- 主导航按客户任务心智组织
- 任务详情中保留足够强的调度、运行记录和异常处理能力
- 采集资产和原始事实作为支持视图存在，不作为主入口压过任务流

## 5. 一期产品信息架构

### 5.1 一级导航

一期建议仅保留以下五个一级模块：

1. `Dashboard`
2. `检查任务`
3. `采集资产`
4. `系统接入`
5. `运行结果`

### 5.2 Dashboard

`Dashboard` 面向客户提供业务化概览，不直接展示大量底层技术术语。建议包含：

- 已接入系统数
- 可用检查任务数
- 今日运行次数
- 今日失败次数
- 最近异常任务
- “新建检查任务”快捷入口

### 5.3 检查任务

这是一期的第一优先级模块，负责承载客户的核心价值。

建议包含：

- 任务列表
- 任务创建向导
- 任务详情
- 调度启停
- 立即运行
- 历史执行查看

### 5.4 采集资产

`采集资产` 负责向客户展示系统中已经可消费的检查对象，但默认以业务友好方式展示。

默认视角：

- 系统
- 页面
- 检查项

详情页中再下钻展示：

- 原始菜单事实
- 页面事实
- 元素事实
- 定位证据
- 资产版本信息

### 5.5 系统接入

`系统接入` 一期只做受控接入表单与状态反馈，不把复杂平台治理逻辑暴露给客户。

建议支持：

- 基础系统信息录入
- 认证方式选择
- 接入状态展示
- 失败原因摘要

### 5.6 运行结果

`运行结果` 作为独立模块，帮助客户跨任务查看检查输出与失败摘要。

建议支持：

- 最近运行记录
- 成功/失败筛选
- 单次执行详情查看
- 按任务、系统维度过滤

## 6. 关键用户流设计

### 6.1 主流程

一期最关键的用户路径定义如下：

`登录 -> Dashboard -> 新建检查任务 -> 选择系统 -> 选择页面/检查项 -> 配置运行参数 -> 设置调度 -> 创建成功 -> 查看任务详情与结果`

这个路径确保用户从一开始就围绕“我要创建一个可运行的检查任务”思考，而不是先理解平台内部实现。

### 6.2 系统接入流

系统接入建议采用受控状态流：

1. 待提交
2. 接入处理中
3. 认证处理中
4. 采集中
5. 资产准备中
6. 可用于创建任务
7. 接入失败

前端在失败态仅输出客户可理解的信息，例如：

- 认证失败
- 页面采集失败
- 资产准备失败
- 需要人工检查

不向客户暴露底层执行实现细节。

## 7. 检查任务向导设计

### 7.1 设计原则

任务向导是一期产品核心入口，必须做到：

- 把底层对象映射成业务语言
- 引导用户一步一步完成配置
- 尽可能少暴露技术实现细节

### 7.2 向导步骤

建议拆为五步：

#### 第一步：选择系统

- 仅展示已可用系统
- 若系统尚未接入，提供“去接入系统”快捷入口
- 若系统仍在准备中，则提示不可继续

#### 第二步：选择检查目标

以业务友好的方式展示：

- 系统
- 页面
- 检查项

在高级信息区可只读显示底层标识：

- `page_asset`
- `page_check`
- 版本标识

#### 第三步：配置运行参数

一期建议支持：

- 严格度
- 时间预算
- 是否先试运行

#### 第四步：设置调度

一期优先采用预设模板，而不是强迫客户编写 cron：

- 每 30 分钟
- 每小时
- 每天
- 工作日每天

仅在高级模式下开放 cron 输入。

#### 第五步：确认与创建

汇总展示：

- 所属系统
- 页面
- 检查项
- 运行参数
- 调度设置

创建成功后进入任务详情页。

## 8. 任务列表与详情设计

### 8.1 任务列表页

建议展示字段：

- 任务名称
- 所属系统
- 检查目标
- 调度状态
- 最近运行状态
- 最近运行时间

建议操作：

- 启用
- 停用
- 立即运行
- 编辑调度
- 查看结果
- 查看资产来源

### 8.2 任务详情页

建议按四个页签组织：

1. `概览`
2. `调度`
3. `运行记录`
4. `资产来源`

其中：

- `概览` 展示任务基础信息、当前状态和最近结果
- `调度` 展示周期、启停状态和下次运行时间
- `运行记录` 展示历史执行与失败摘要
- `资产来源` 展示绑定的页面、检查项、资产版本，以及原始事实查看入口

## 9. 前端与后端能力映射

### 9.1 产品对象与底层对象映射

一期前端建议定义以下产品对象：

1. `系统`
2. `检查任务`
3. `采集资产`
4. `运行结果`

其中映射关系如下：

#### 系统

面向客户展示接入对象，底层主要关联：

- `system`
- `system_auth_policy`
- `system_crawl_policy`

#### 检查任务

这是产品层对象，不直接等同于单个底层表。底层典型映射为：

- 选择阶段命中 `page_asset -> page_check`
- 创建调度后生成 `published_job`
- 手动运行触发 `run_page_check` 或触发 `published_job`
- 结果查看关联 `check_request` 与 `job_run`

#### 采集资产

默认面向客户展示页面与检查项，详情中再映射到底层：

- 菜单事实
- 页面事实
- 元素事实
- 资产版本

#### 运行结果

用于聚合任务与执行记录，不直接暴露底层运行复杂度。

### 9.2 前端边界

前端必须遵守以下边界：

- 不绕过 `control_plane`
- 不直接处理服务端认证注入
- 不输出完整 `storage_state`
- 不把脚本渲染结果当作客户主模型

## 10. API 与读模型建议

### 10.1 设计原则

一期前端不宜直接把后端现有 schema 原样暴露到页面层，而应提供面向页面的聚合读模型。

### 10.2 建议补充的前端友好接口

为避免前端自行拼接过多请求，建议后端补充少量聚合查询接口：

1. `系统接入视图接口`
   - 返回系统基础信息、当前接入状态、失败摘要、最近处理时间

2. `任务向导资产浏览接口`
   - 按系统返回页面、检查项、可用状态与简要说明

3. `任务详情聚合接口`
   - 返回任务基础信息、调度配置、最近运行、资产来源

4. `运行结果聚合接口`
   - 返回按任务/系统可过滤的执行记录视图

### 10.3 不建议的做法

不建议前端自己直接拼装完整正式链路，例如：

- 先调 `auth refresh`
- 再调 `crawl`
- 再调 `compile`
- 再猜测何时可创建任务

更合理的方式是由后端提供受控接入入口和聚合状态视图。

## 11. 前端技术选型

### 11.1 选型结论

一期技术选型确定如下：

- `React + Vite`
- 不使用重型后台脚手架
- 采用轻量自建管理台

### 11.2 选择 React + Vite 的原因

1. 与当前仓库 `README` 中声明的前端技术栈保持一致
2. 避免未来出现 React/Vue 双栈维护成本
3. 更适合围绕任务向导、资产详情、运行记录等领域流程构建定制化页面

### 11.3 不使用重型脚手架的原因

本项目的一期前端是“领域流程型产品界面”，而不是通用 CRUD 后台。使用重型脚手架会带来以下问题：

- 过度预置传统 admin 的导航与权限模型
- 对任务向导、状态流、资产详情等页面形态束缚较大
- 后续定制成本高于前期收益

### 11.4 推荐的前端技术边界

建议采用：

- React + Vite
- Router
- 轻量状态管理
- UI 组件库
- 面向页面的 API 封装

避免在一期引入：

- 复杂微前端
- 重度代码生成
- 与现有项目割裂的独立前端架构体系

---

## 12. 多团队/多客户扩展预留

虽然一期不做复杂多租户，但信息架构需要预留扩展边界：

1. 登录后保留“当前工作空间/团队”概念占位
2. 产品对象避免按个人视角硬编码
3. 任务、系统、运行结果视图要可自然扩展到组织维度筛选
4. 页面结构避免把权限模型写死为单管理员后台

---

## 13. 验收口径

一期前端设计完成后，应以以下结果作为验收标准：

1. 客户可以登录并进入统一管理台
2. 客户可以提交基础系统接入信息并看到接入状态
3. 客户可以从业务友好资产视图中选择检查项并完成任务创建
4. 客户可以为任务设置定时运行并执行启停
5. 客户可以查看最近运行结果与失败摘要
6. 客户可以在详情页查看原始采集事实，而不影响主流程可理解性

---

## 14. 实施建议

前端实施顺序建议如下：

1. 先完成登录壳、主布局与导航结构
2. 再完成“检查任务”主流程与任务详情
3. 然后补“系统接入”与接入状态页
4. 再补“采集资产”视图与原始事实详情
5. 最后补“运行结果”聚合视图与细节打磨

这一顺序与当前项目“先资产模型和控制面，再补更复杂能力”的工程原则保持一致。

---

## 15. 最终结论

一期 Web 前端管理平台应定位为：**面向客户侧普通使用者的任务中心型管理后台**。

其核心结论如下：

- 主目标是“配置检查任务并定时运行”
- 主心智是“检查任务”，而不是底层 `page_check / published_job`
- 一级导航收敛为：
  - `Dashboard`
  - `检查任务`
  - `采集资产`
  - `系统接入`
  - `运行结果`
- 技术选型确定为：
  - `React + Vite`
  - 轻量自建
  - 不使用重型后台脚手架
- 正式执行、认证注入、资产编译与调度仍统一依赖现有后端控制面

该方案既能满足客户一期可用，也为后续多团队、多客户、更多治理能力扩展留下清晰边界。
