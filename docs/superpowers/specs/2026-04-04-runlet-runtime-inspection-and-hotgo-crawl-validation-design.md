# Runlet 联调验收与 HotGo 采集校验设计

**日期：** 2026-04-04  
**作者：** Codex  
**状态：** Draft

---

## 1. 文档定位

本文档定义一次面向当前 Runlet 项目的真实联调验收方案，目标不是新增功能，而是以真实服务、真实前端浏览器、真实外部被采集系统为基础，验证当前平台在本地环境中的可运行性、接口可用性、前端可用性，以及针对 `hotgo` 系统的采集与资产编译正确性。

本次设计严格遵守现有平台边界：

- 检查资产是主模型
- Playwright 脚本是派生产物
- 正式执行统一走 `control_plane`
- 认证注入必须由服务端统一处理

因此，本次工作不是单独验证一个脚本是否可跑，而是验证平台本身的正式链路是否真实可用。

---

## 2. 背景与目标

### 2.1 已确认范围

本次联调验收覆盖以下四个验证面：

1. Runlet 自身后端服务启动是否正常
2. Runlet 自身前端启动与联调是否正常
3. Runlet 平台核心 API 调用是否存在报错
4. 基于数据库里已有的 `hotgo` 系统记录，真实触发一次新的 `crawl -> asset compile`，并检查菜单、页面、关键元素、检查资产四层结果是否完整正确
5. Runlet 平台前端页面必须做一次性真实 Playwright 巡检，不以单元测试替代

### 2.2 已确认边界

本次验证边界已经明确如下：

- 不检查另外两个外部测试系统，只聚焦 `hotgo`
- `hotgo` 使用数据库中已有的系统记录，不新建临时接入对象
- 允许实际触发新的 `crawl -> asset compile`
- 前端 Playwright 只做一次性临时真实浏览器巡检，不把 E2E 配置和测试用例落仓库

### 2.3 本次工作的产出目标

本次工作最终要形成一份基于真实运行证据的结论，回答以下问题：

- 当前后端 API、worker、scheduler 是否能正常启动
- 当前前端是否能正常启动并完成基本联调
- 当前平台核心接口是否存在 4xx/5xx 或结构性错误
- 当前平台主要管理页面是否存在白屏、路由错误、加载失败或关键功能不可用
- 当前 `hotgo` 的采集与资产编译结果是否真实、完整、可消费

---

## 3. 非目标

本次联调验收明确不做以下事项：

- 不为本次巡检建设长期维护的前端 E2E 基础设施
- 不对三个测试系统做横向对比巡检
- 不把本次工作扩展为大规模自动化测试体系建设
- 不直接修改 `hotgo` 业务系统页面结构或账号配置
- 不绕过 `control_plane` 直接拼接自由执行脚本验证正式链路

---

## 4. 方案比较与推荐

### 4.1 方案 A：轻量烟雾巡检

做法：

- 启动本地服务
- 做少量 API 调用
- 对前端做少量页面打开验证
- 对 `hotgo` 只确认能够触发采集

优点：

- 执行速度快
- 能快速发现明显启动故障

缺点：

- 不能证明平台主链路真实可用
- 无法确认 `hotgo` 采集内容的完整性与正确性

### 4.2 方案 B：验收型联调检查

做法：

- 启动 `backend API + worker + scheduler + front`
- 对平台核心 API 做真实调用
- 对平台前端做真实 Playwright 巡检
- 对已有 `hotgo` 记录触发新的 `crawl -> asset compile`
- 再用真实浏览器登录 `hotgo` 做结果对照

优点：

- 能同时覆盖平台运行、平台联调和外部采集真实性
- 结论与用户当前诉求一致
- 证据链清晰，可直接用于验收

缺点：

- 耗时高于基础烟雾检查
- 需要同时处理本地服务与外部系统访问稳定性

### 4.3 方案 C：全量深度巡检

做法：

- 在方案 B 的基础上，尽量遍历更多平台 API、更多前端页面，以及 `hotgo` 更多菜单页面

优点：

- 覆盖最全
- 有利于形成更接近回归测试的数据面

缺点：

- 成本高
- 容易把一次性验收扩展成长期测试体系建设

### 4.4 推荐结论

采用 **方案 B：验收型联调检查**。

原因：

- 它既能验证 Runlet 平台本身是否可运行，也能验证 `hotgo` 采集与资产编译是否真实有效
- 覆盖面足以支撑一次真实验收
- 不会把当前任务膨胀成新的测试工程建设

---

## 5. 验收对象与验证维度

### 5.1 平台运行维度

验证对象：

- `backend API`
- `worker`
- `scheduler`
- `front`

关注点：

- 能否启动
- 启动时是否有配置、迁移、依赖、连接类错误
- 启动后核心请求是否立即失败

### 5.2 平台 API 维度

验证对象：

- 控制台认证接口
- 门户概览接口
- 任务、资产、系统、结果相关接口
- 与 `hotgo` 采集、编译、结果读取有关的控制面接口

关注点：

- 状态码是否正常
- 返回结构是否满足前端消费
- 是否存在 4xx/5xx、序列化问题、鉴权问题或结构不一致问题

### 5.3 平台前端维度

验证对象：

- 登录页
- `Dashboard`
- `检查任务`
- `采集资产`
- `系统接入`
- `运行结果`

关注点：

- 页面是否可访问
- 是否存在白屏、崩溃、长时间 loading、关键区块缺失
- 页面跳转、列表加载、详情页打开等主交互是否可用

### 5.4 HotGo 采集真实性维度

验证对象：

- 菜单事实
- 页面事实
- 关键元素事实
- 资产编译产物

关注点：

- 菜单是否采全
- 关键页面与路由是否入库
- 关键元素是否被稳定采集，定位策略是否符合约束
- 是否编译出 `page_asset`、`page_check`、`module_plan`

## 6. 执行流程设计

### 6.1 第一步：环境与服务启动检查

执行顺序：

1. 检查 `backend/.env`、数据库、Redis、本地依赖与端口占用
2. 启动 `backend API`
3. 启动 `worker`
4. 启动 `scheduler`
5. 启动 `front`

记录内容：

- 每个服务的启动日志
- 启动阶段的首个异常
- 端口监听状态
- 首个核心请求的返回结果

### 6.2 第二步：平台 API 实调

优先验证以下接口族：

- `/api/console/auth/login`
- `/api/console/auth/me`
- `/api/console/portal/dashboard`
- `/api/console/portal/systems`
- `/api/console/tasks/`
- `/api/console/assets/`
- `/api/console/results/`

随后验证与 `hotgo` 相关的控制面接口，包括：

- 定位 `hotgo` 系统记录
- 触发新一轮 `crawl`
- 跟踪 `asset compile`
- 读取新的资产与检查结果

### 6.3 第三步：平台前端 Playwright 巡检

巡检方式：

- 真实打开本地前端地址
- 使用平台控制台账号登录
- 依次访问主要页面
- 观察控制台、网络请求、页面结构与关键交互

最小覆盖路径：

1. 登录页
2. Dashboard
3. 检查任务列表
4. 采集资产列表
5. 系统接入列表
6. 运行结果列表

必要时补充：

- 详情页打开
- 页面内跳转
- 空态/报错态显示

### 6.4 第四步：触发 HotGo 新采集与资产编译

执行要求：

- 使用数据库里已有的 `hotgo` 系统记录
- 通过平台正式接口触发新的 `crawl`
- 由 worker 消费任务并自动串起 `asset compile`
- 不走自由脚本旁路

记录内容：

- crawl job 状态变化
- compile job 状态变化
- 失败信息或汇总结果
- 新增 `snapshot/asset/check` 的数量与关键信息

### 6.5 第五步：HotGo 真实页面对照验证

对照方式：

- 用 Playwright 直接登录 `hotgo`
- 读取真实左侧菜单和关键页面
- 与 Runlet 中本轮采集得到的事实层与资产层结果进行对照

重点验证四层：

1. 菜单层：左侧导航是否基本覆盖
2. 页面层：关键菜单对应页面与路由是否落库
3. 元素层：关键交互元素是否采到，且优先使用稳定定位
4. 资产层：关键页面是否形成 `page_asset/page_check/module_plan`

### 6.6 第六步：结果归档与结论输出

最终报告按以下维度组织：

- 服务启动问题
- 平台 API 问题
- 平台前端问题
- `hotgo` 采集完整性与正确性问题
- 总体结论

## 7. 验收标准与失败判定

### 7.1 服务启动健康

通过标准：

- API 能启动并正常响应核心请求
- worker 能启动并持续轮询
- scheduler 能启动并进入装载循环
- front 能启动且代理可访问后端

失败判定：

- 启动报错、端口冲突、依赖缺失、迁移失败、数据库或 Redis 无法连接
- 启动成功但首个核心请求即失败

### 7.2 平台 API 可用性

通过标准：

- 核心接口返回正常结构
- 关键读接口与至少一条关键写接口可用
- 不出现非预期 4xx/5xx

失败判定：

- 核心接口 5xx
- 接口结构不稳定，导致前端不可消费
- 鉴权、序列化或业务状态导致主流程中断

### 7.3 平台前端可用性

通过标准：

- 可登录
- 主页面可访问
- 首屏无白屏、崩溃、无限加载
- 核心交互可完成

失败判定：

- 页面白屏、报错、路由崩溃
- 接口异常导致页面核心区块不可用
- 关键流程无法进入下一步

### 7.4 HotGo 采集正确性

通过标准：

- `crawl -> asset compile` 正常完成
- 菜单、页面、关键元素、检查资产四层结果基本完整
- 关键元素定位证据符合项目稳定定位约束

失败判定：

- crawl 或 compile 任一步失败
- 明显漏采菜单、页面或关键元素
- 未产出可消费资产
- 采集结果与真实页面偏差过大，导致资产不可用于正式检查

## 8. 证据采集策略

为了保证本次结论可追溯，执行过程中需要保留以下证据：

- 本地服务启动日志
- 关键 API 调用及状态码、核心响应字段
- 前端页面截图
- 浏览器控制台或网络错误摘要
- `hotgo` 真实页面截图
- 采集后事实层与资产层关键数据摘要
- crawl 与 compile 的作业状态及结果摘要

---

## 9. 风险与边界处理

### 9.1 外部系统波动

`hotgo` 为外部被采集系统，可能存在网络波动、验证码波动或页面临时变化。本次验证需要把“平台故障”和“外部系统波动”分开记录，避免误归因。

### 9.2 本地环境差异

数据库迁移状态、Redis 状态、前端依赖、Playwright 浏览器依赖都可能影响结果。执行前应优先做本地基础检查，避免把环境缺陷误记为业务缺陷。

### 9.3 一次性浏览器巡检的边界

本次 Playwright 前端巡检是一次性真实巡检，其结论可用于当前验收，但不等同于后续可持续回归测试能力。

---

## 10. 预期交付物

本次联调验收完成后，输出应至少包括：

1. 一份真实运行报告
2. 一份按严重程度整理的问题清单
3. 每个问题的证据来源说明
4. 对 Runlet 当前状态的总体判断：
   - 可用
   - 部分可用
   - 不可用

---

## 11. 结论

本次工作应以“验收型联调检查”为主，而不是轻量烟雾测试或长期 E2E 建设。执行时要同时覆盖平台运行、平台前后端联调、平台核心 API 和 `hotgo` 真实采集正确性四个面，最终给出基于真实服务、真实浏览器和真实外部系统的证据化结论。
