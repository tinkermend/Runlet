# 当前态语料切换与历史归档模型设计

**日期：** 2026-04-06  
**作者：** Codex  
**状态：** Draft

---

## 1. 文档定位

本文档定义 Runlet 采集事实层从“历史快照累积模型”收敛到“当前态语料模型”的设计，目标是在不改变 `control_plane -> page_asset/page_check -> runner_service` 正式执行主链的前提下，解决采集事实层长期累积导致的数据污染、查询歧义与无意义 DML 问题。

本文档只讨论以下范围：

- `crawl_snapshots / pages / menu_nodes / page_elements` 的当前态、候选态、历史态组织方式
- 新采集的候选写入、差异比较、验证、切换与历史归档规则
- 事实层变化触发条件、删除页面判定条件与正式切换事务边界

本文档不讨论以下范围：

- `runner_service` 的模块执行能力重构
- 自然语言解析主模型从 `page_asset/page_check` 迁移到事实层
- Playwright 脚本产物模型变更
- 合规审计、长期历史归档保留策略之外的冷数据治理

---

## 2. 背景与问题

### 2.1 当前问题

当前采集链路会为每次 crawl 创建一个新的 `crawl_snapshot`，并将本次发现的 `pages / menu_nodes / page_elements` 全量写入数据库。该设计天然允许同一系统在多个 `snapshot_id` 下长期保留多份相同或近似相同的页面事实。

这带来三个直接问题：

1. **事实查询混入历史数据。** 直接查询 `pages` 或 `menu_nodes` 时，如果不显式限定 `snapshot_id`，很容易把历史页面和当前页面混在一起。
2. **事实层产生“看似污染”的重复数据。** 同一路径、同一标题、同一菜单标签会在数据库中长期保留多份记录，但这些旧记录对当前自动化仿真测试并无直接价值。
3. **无意义的数据库写入累积。** 即使采集结果与当前正式语料完全一致，系统仍会继续落一批新的事实记录，增加在线库负担。

### 2.2 当前模型与业务真实需求不一致

Runlet 当前数据库中的采集事实，本质上是供资产编译与自动化仿真测试使用的“当前有效语料”。一旦目标系统菜单、页面结构或关键元素发生变化，旧事实对主链就失去实际业务价值。

因此当前需求并不是“保留全部历史快照以供在线查询”，而是：

- 在线库只保留当前正式语料
- 新采集先写候选集
- 只有高质量且确有变化时才切换正式集
- 被替换掉的旧正式集完整归档到冷历史区

### 2.3 业务边界

本项目的执行主模型仍必须保持为：

- 检查资产是主模型
- Playwright 脚本是派生产物
- 正式执行统一走 `control_plane`
- 认证注入必须由服务端统一处理

因此本次改造只能调整事实层的存储与切换模型，不能让正式执行回退为直接消费 `pages/menu_nodes/page_elements`。

## 3. 目标、非目标与约束

### 3.1 目标

1. 让在线事实层默认只服务“当前有效语料”。
2. 支持“先写新数据，验证成功后原子切换并删除旧正式数据”。
3. 只有在事实层出现有效语义变化时，才执行正式切换与历史归档。
4. 在旧数据唯一且正确时，不进行无意义 DML。
5. 只有在高质量、非降级采集结果下，才允许把旧页面判定为已删除。
6. 被替换掉的旧正式集完整归档到 `*_hist` 冷数据表，便于排障追溯。

### 3.2 非目标

1. 不把历史表纳入默认线上业务查询链路。
2. 不要求保留每次采集都产生一份历史副本。
3. 不保留低质量候选或无差异候选的长期事实记录。
4. 不在本次设计中展开资产层历史版本治理细节。

### 3.3 约束

1. 文档与实现边界必须保持中文表述。
2. 正式执行主链仍以 `intent_aliases -> page_assets -> page_checks` 为主。
3. 删除页面必须基于“高质量且非降级”的采集结果，而不能基于“本次没采到”直接推断。
4. 历史归档只发生在正式集发生有效变化并成功切换时。

---

## 4. 方案比较

### 4.1 方案 A：保留当前快照模型，只在查询层补约束

做法：

- 继续保留多份 `crawl_snapshots/pages/menu_nodes/page_elements`
- 通过 API 和 SQL 约束默认只读最新 `snapshot`

优点：

- 改动最小
- 对当前采集代码侵入较低

缺点：

- 正式表里仍长期堆积历史事实
- 数据污染感不会消失
- 仍依赖查询者始终记得加“最新快照”过滤条件

### 4.2 方案 B：当前态、候选态、历史态三分模型（推荐）

做法：

- 在线正式区只保留一个 `active` 采集批次
- 每次新采集先写入一个 `draft` 候选批次
- 高质量且存在事实层有效变化时，归档旧正式集到 `*_hist`，再原子切换到新正式集
- 无变化或低质量候选直接丢弃

优点：

- 与“当前有效语料仓”目标一致
- 在线查询天然不会混入历史
- 能支持“先写候选、验证后切主、旧正式完整归档”

缺点：

- 需要明确新增候选态/正式态管理与历史归档链路
- 需要切换事务和差异比较逻辑

### 4.3 方案 C：取消批次概念，直接对当前态表增量更新

做法：

- 不再写 `draft` 批次
- 直接在当前态表上做逐行 upsert / delete

优点：

- 表面模型最简单

缺点：

- 无法自然支持“先写新数据、验证通过后切换”
- 删除判定和失败回滚风险更高
- 很难保证事实层切换的原子性

### 4.4 推荐结论

采用方案 B。

原因：

- 它最符合 Runlet 当前“在线库只保留当前有效语料”的业务目标。
- 它能保留候选比较与正式切换的安全边界。
- 它把历史归档从在线主数据中剥离，既满足排障追溯，又避免污染正式查询。

## 5. 总体设计

### 5.1 三类数据区

事实层划分为三个语义区：

- **正式区（active）**
  - 当前线上有效语料
  - 默认供资产编译、控制台事实展示、诊断查询使用
- **候选区（draft）**
  - 本次新采集候选结果
  - 仅用于验证、比较、切换
- **历史区（hist）**
  - 被正式切换替换掉的旧正式集完整归档
  - 默认不参与线上主查询

### 5.2 批次模型

保留 `crawl_snapshots` 作为采集批次头表，但其语义收敛为“当前态/候选态管理头表”，而不是长期在线历史档案。

建议至少具备以下状态：

- `draft`
- `active`
- `discarded`

同时约束：

- 同一系统任一时刻最多只有一个 `active snapshot`
- 任一新采集只能先写入新的 `draft snapshot`

### 5.3 默认读路径

所有默认事实层读取口径都必须只针对当前 `active snapshot`：

- 控制台事实查看
- 资产编译输入
- 事实诊断接口

历史查询必须走独立冷数据诊断入口，不允许默认 join `*_hist`。

---

## 6. 数据模型设计

### 6.1 在线正式/候选模型

保留当前在线事实表：

- `crawl_snapshots`
- `pages`
- `menu_nodes`
- `page_elements`

其中：

- `pages / menu_nodes / page_elements` 继续通过 `snapshot_id` 隶属于某个批次
- 同一 `snapshot_id` 下代表一份完整的候选或正式事实集

### 6.2 历史归档模型

新增完整冷归档表：

- `crawl_snapshots_hist`
- `pages_hist`
- `menu_nodes_hist`
- `page_elements_hist`

历史表原则：

- 完整保留旧正式集的全部原始字段
- 不做字段裁剪，不只保留摘要
- 以批次整体归档，保持旧正式集的结构完整性

建议增加最少归档元信息：

- `archived_at`
- `archive_reason`
- `source_active_snapshot_id`
- `replaced_by_snapshot_id`

### 6.3 唯一性与完整性要求

正式区必须补充明确约束：

1. 同一系统最多一个 `active snapshot`
2. 同一正式批次内 `pages.route_path` 唯一
3. 同一正式批次内菜单节点身份必须可重复推导，避免父链混乱
4. 默认资产层当前态对象仍保持唯一命中语义

这些约束是“旧数据唯一且正确则不处理”的前提。

---

## 7. 页面语义指纹与差异判定

### 7.1 差异判定原则

只有当事实层出现**有效语义变化**时，才允许进入正式切换和历史归档。

“有效语义变化”不以单纯行数变化为准，而以页面级语义指纹为准。

### 7.2 页面语义指纹组成

建议每个页面生成一个页面语义指纹，至少包含：

- `route_path`
- `page_title`
- 菜单结构或菜单链摘要
- 对自动化检查有意义的关键元素摘要
  - 例如表格、查询区、按钮、分页、标签页等
- 必要的状态上下文摘要

### 7.3 比较规则

候选 `draft` 与当前 `active` 比较时：

- 页面集合一致
- 每个页面语义指纹一致
- 菜单结构无差异
- 关键元素摘要无差异

则视为**无变化**。

一旦出现以下任一情况，则视为**有效变化**：

- 新增页面
- 页面语义指纹变化
- 菜单挂载关系变化
- 关键元素集合变化
- 页面删除

### 7.4 无变化处理

若候选集与当前正式集语义等价，则：

- 丢弃 `draft`
- 不归档历史
- 不切换正式集
- 不执行正式区 DML

这条规则是避免“旧数据唯一且正确时仍频繁更新数据库”的核心机制。

---

## 8. 删除页面判定

### 8.1 风险边界

“本次采集没有采到某页面”并不等于“该页面真的已被删除”。采集不完整、菜单未展开完全、认证退化或页面降级都可能导致候选集缺页。

### 8.2 删除成立条件

只有在候选采集结果满足以下条件时，才允许把旧正式页面判定为已删除：

- `quality_score` 达到阈值
- `degraded = false`

本文档将其视为“高质量、非降级采集”。

### 8.3 删除规则

若候选采集高质量且非降级，并且：

- 当前正式集存在某页面
- 候选集不存在对应页面

则允许在正式切换时将其视为“已删除页面”。

若不满足高质量、非降级条件，则：

- 即使候选集缺页，也不能删除旧正式页面
- 本次候选只能被丢弃或判定为切换失败

---

## 9. 切换与归档流程

### 9.1 采集阶段

每次采集执行如下步骤：

1. 创建新的 `draft snapshot`
2. 将本次 `pages/menu_nodes/page_elements` 写入该 `draft`
3. 不影响当前 `active` 正式数据

### 9.2 验证阶段

切换前必须进行三类验证：

1. 质量验证  
   `quality_score` 达标且 `degraded = false`

2. 结构验证  
   候选集内 `route_path` 唯一且结构可用

3. 差异验证  
   比较 `draft` 与当前 `active` 的页面语义指纹

### 9.3 验证结果分支

验证结果分三类：

- **无变化**
  - 丢弃 `draft`
  - 不归档
  - 不切换

- **有变化但低质量**
  - 丢弃 `draft`
  - 不归档
  - 不切换

- **有变化且高质量**
  - 进入正式切换事务

### 9.4 原子切换事务

推荐在单个数据库事务内完成以下步骤：

1. 锁定当前系统的 `active snapshot`
2. 将旧 `active` 对应的 `crawl_snapshots/pages/menu_nodes/page_elements` 全量复制到 `*_hist`
3. 删除旧 `active` 对应的在线正式事实数据
4. 将当前 `draft snapshot` 状态改为 `active`
5. 让事实层默认查询口径切换到新的 `active snapshot`
6. 提交事务

这样可以保证：

- 不会出现“旧正式已删、新正式未切上”的中间态
- 历史归档与正式切换保持原子一致
- 在线主链始终只看到一套正式语料

### 9.5 故障回滚

若切换事务任一环节失败，则：

- 整个事务回滚
- 旧 `active` 保持不变
- `draft` 维持失败候选或改为 `discarded`
- 不允许出现部分归档、部分切换成功

## 10. 对资产层与 API 的影响

### 10.1 资产层边界

本次改造不改变正式执行主模型：

- 运行时仍默认从 `intent_aliases -> page_assets -> page_checks` 命中
- `page_asset/page_check` 仍是正式执行入口
- `runner_service` 仍只消费被批准的计划

### 10.2 资产层更新时机

事实层正式切换成功后，资产层应基于新的 `active snapshot` 重新编译当前态资产。

本文档不强制规定资产层是否同步引入 `*_hist`，但明确要求：

- 当前态资产只应服务当前正式事实
- 不允许继续让历史事实污染默认资产命中链路

### 10.3 API 查询口径

默认 API 必须显式只读当前正式态：

- 控制台事实查询只读 `active snapshot`
- 资产编译输入只读 `active snapshot`
- 冷历史查询必须走独立诊断接口

不允许继续依赖“全表事实 + 调用方自行过滤”的模式。

---

## 11. 测试策略

至少补以下验证：

1. 新采集与当前正式集无差异时，`draft` 被丢弃且不发生历史归档。
2. 新采集有差异且高质量时，旧正式集被完整归档，新正式集成功切换。
3. 新采集有差异但 `degraded=true` 时，不允许切换，也不允许删除旧页面。
4. 候选集缺失某页面，但 `quality_score` 不达标时，不允许将旧页面判定为删除。
5. 同一系统同时只能存在一个 `active snapshot`。
6. 正式切换失败时，归档与切换整体回滚。

---

## 12. 风险与控制

主要风险如下：

1. **差异判定过宽，导致频繁切换。**  
   需要把“有效变化”严格约束为页面语义指纹变化，而不是任意事实波动。

2. **删除判定过激，误删正式语料。**  
   必须坚持“只有高质量、非降级采集才允许删除页面”的规则。

3. **切换事务不完整，产生正式态中间状态。**  
   必须把归档、删除旧正式、切换新正式放在同一事务中。

4. **历史表重新污染主链。**  
   必须把 `*_hist` 明确隔离为冷数据区，默认不进入主查询。

---

## 13. 结论

Runlet 的采集事实层不应继续作为长期在线历史快照仓，而应收敛为“当前态语料仓 + 候选验证区 + 冷历史归档区”的三分模型。

该模型的核心规则是：

- 在线主链只面对当前正式语料
- 新采集先写候选
- 高质量且存在事实层有效变化时才切换
- 无变化或低质量候选直接丢弃
- 被替换掉的旧正式集完整归档到历史区

这样既能消除当前事实层重复与查询歧义问题，又能保留企业级排障所需的完整历史副本，同时避免在“旧数据唯一且正确”时持续制造无意义数据库写入。
