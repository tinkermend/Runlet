当前 我构建了一个基于AI的 智能web仿真系统, 系统整体设计逻辑如下,请根据我的问题点帮我深入分析共同探讨更好的实现方案:

- 通过playwright 自动定时登录需要爬取的web 系统网站,获取登录 StorageState与token 信息并保存在数据库中
- 通过自动注入认证信息+playwright脚本 爬取网站 菜单,页面路由数据,以结构化树形方式存储在数据库中
- 通过编写mcp 工具暴露给其他AI工具调用使用获取抓取的数据用于构建 playwrigh 仿真测试脚本
- 通过skills中调用mcp 和编排来实现 当用户说" 看一下xx系统xx页面是否正常",自然语言解析->mcp 数据获取,动态生成playwright脚本能力
- 数据库里面存储的逻辑树按 `系统 -> 页面 -> 元素` 定义
- 当前项目默认只支持 vue,react 框架的web平台

但我现在发现我构建的这个技术框架存在以下几个问题:

- 1.mcp 结构再做细化返回的数据也非常多, 因为很难打通 用户描述->数据获取的 精细化控制

- 3.如果想把生成的脚本进行固化, 但因为实现上是通过mcp 获取登录认证数据再注入了, 固化的脚本很难确保生成每次都会调用mcp 进行获取,并且安全性上面也很难保证

- 2. 将一些不会变更的操作封装成模块, 通过模块拼接来实现快速playwright 脚本生成

1. 原始采集仓
   继续保留你现在的 系统 -> 菜单 -> 页面 -> 元素 结构，但它只负责“事实记录”，不再直接作为运行时给 LLM 的主输入。
   这里存：

系统配置、认证方式、最新有效 storage_state
菜单树、页面路由、元素定位器
采集时间、稳定度、框架类型、质量分

这层回答的是“系统现在长什么样”。

2. 执行资产仓
   在原始采集仓之上新增一层“编译结果”。
   资产分两类：

页面级资产：如“进入用户管理页并确认页面可打开”“列表成功渲染”“新增弹窗可打开”
模块级动作：如“注入认证”“按菜单链导航”“等待表格稳定”“打开新增弹窗”
页面级资产是运行主单元，模块级动作是生成底座和复用单元。用户请求不直接命中元素树，而是优先命中页面级资产。

这层回答的是“这个系统可以怎么被稳定执行”。

3. 统一执行面
   新增服务 API 作为唯一正式执行入口。
   Skills、MCP、CLI 都不要再直接拼 Playwright 脚本执行真实检查，而是统一调用服务 API，由服务端完成：

意图归一化
资产命中
认证注入
执行计划选择
结果产出
失败后的补采集/补编译触发
这层回答的是“这次检查应该怎么跑”。

4. MCP 和 CLI 的新定位

MCP：只做轻量检索、发现、解释，不做主执行编排
CLI：只做运维、调试、回灌、手工验证
Skills：只负责把自然语言请求转成服务 API 的受控调用

基础闭环：2026-04-01-ai-playwright-platform-foundation.md
认证与采集：2026-04-01-ai-playwright-platform-auth-crawl.md
资产编译与漂移：2026-04-01-ai-playwright-platform-asset-compiler.md
Runner、脚本渲染与调度：2026-04-01-ai-playwright-platform-runner-scheduling.md
这四份计划对应四个独立可交付阶段，边界是刻意切开的：

第一份只建平台骨架和 control-plane 基线
第二份才进入真实 auth/crawl
第三份才做 asset compiler 和 drift
第四份才做 runner、script render、published jobs 和 scheduler
